# 10. 多渠道支持

## 本章定位

在 L1（Harness 101）阶段，我们**只完成设计预留，不实现完整多渠道**。

原因：

- 核心 Harness（Agent / Memory / Session / Skills / Tools）需要先稳定
- 多渠道属于「接入层」，不应过早耦合进核心
- 保持 L1 可跟做、体量可控

本章说明设计思路，真正的 Telegram / REST API / 桌面端实现留给 L2。

## 设计原则

**Core 与 Channel 彻底分离。**

```text
Channels（接入层）
  CLI · Desktop · REST API · Telegram · WeChat · Slack
        │
        ▼  统一消息协议（IncomingMessage / OutgoingMessage）
Core Harness
  Agent · Memory · Session · Skills · Tools
```

任何渠道都只做三件事：

1. 把用户输入转成统一的 IncomingMessage
2. 调用 Core
3. 把 OutgoingMessage 转成渠道自己的展示/发送格式

Core 不知道自己正在被哪个渠道调用。

## 当前已实现

| 渠道 | 状态 |
|------|------|
| CLI | ✅ 已实现（scripts/chat.ts） |
| REST API | ❌ L2 |
| Telegram | ❌ L2 |
| Desktop | ❌ L2 |

CLI 是验证核心逻辑的最佳第一渠道：实现简单、反馈直接、方便调试。

## 预留的扩展点

在代码结构上我们已预留：

```text
packages/
  channels/
    cli/
    api/          # 预留
    telegram/     # 预留
    desktop/      # 预留
apps/
  cli/
  server/         # 预留：同时挂 API + Bot
  desktop/        # 预留
```

后续新增渠道时，只需：

- 实现渠道适配器
- 调用同一个 Agent + Memory + Session
- 不必改 Core 内部逻辑

## 本章总结

多渠道是个人助理的重要能力，但不是 L1 的重点。
L1 我们用 CLI 跑通了完整闭环；L2 再把同一套 Core 接到更多渠道上。
