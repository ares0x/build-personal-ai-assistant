# 11. 权限、安全与可扩展性

## 本章定位

L1 只建立**最小安全意识与扩展结构**，不实现完整的生产级权限系统。

## 为什么需要权限？

一旦 Agent 能执行工具（写文件、调 API、执行命令），就必须回答：

- 哪些操作可以自动执行？
- 哪些需要用户确认？
- 如何防止误操作或提示注入带来的危险调用？

## L1 中的最小实践

当前阶段我们做了这些基础约束：

1. **工具显式注册**
   只有通过 `ToolRegistry.register` 的工具才能被调用。

2. **Skills 显式注册**
   模型只能「看到」我们注入的技能列表，而不是任意发挥。

3. **系统提示中的行为规则**
   明确告诉模型：只能使用提供的技能和工具。

4. **本地数据隔离**
   记忆和会话存在本地 `data/`，默认不上传。

这些还远不够生产使用，但已经建立了「默认不信任、显式授权」的方向。

## L2 需要补强的方向

| 方向 | 说明 |
|------|------|
| 工具权限分级 | read / write / network / system / dangerous |
| 审批流 | 危险操作需用户确认 |
| 沙箱 | 限制文件与网络访问范围 |
| 审计日志 | 记录每次工具调用 |
| 提示注入防护 | 减少恶意输入劫持 Agent 行为 |

## 可扩展性设计回顾

L1 已落地的扩展点：

- **Provider 模式**：可替换 LLM（Mock / OpenAI / DeepSeek）
- **ToolRegistry**：可继续注册新工具
- **SkillRegistry**：可继续注册新技能
- **Memory / Session 接口**：存储实现可替换（文件 → SQLite → 其他）
- **Channel 与 Core 分离**：为多渠道预留结构

只要保持这些边界清晰，项目可以持续演进而不必推倒重来。

## 本章总结

安全与扩展不是最后再补的「附加项」，而是架构早期就该考虑的约束。

L1 我们用最小方式落实了这些意识；L2 再系统化实现权限模型与更强的扩展机制。
