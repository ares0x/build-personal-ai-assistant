# 01. 项目定位与架构设计

## 我们到底要做什么？

hachimi 的目标不是再做一个「更强的 Coding Agent」，而是做一个：

> **以「个人助理」为核心的通用 Agent Harness**

它需要具备以下能力：

- 越用越懂用户（强长期记忆）
- 支持写作、作图、知识管理、日程等多种任务
- 同时支持 CLI、桌面端、REST API、即时通讯软件
- 架构清晰、可扩展、可维护

简单来说，我们要做的是**底层的 Harness**，而不是某一个具体的应用。

## 为什么不直接用现有产品？

当前主流产品存在明显局限：

| 产品 | 主要问题 |
|------|----------|
| Claude Code | 强绑定编程场景，记忆和扩展能力封闭 |
| Cursor | 编辑器绑定，难以做成多渠道个人助理 |
| Grok Build | 优秀但目前主要面向编程，且刚开源 |
| Pi | 设计极简优秀，但定位仍偏 Coding Agent |

我们需要一个**更通用、更以「人」为中心**的架构。

## 设计原则

1. **Core 与渠道彻底分离**
   核心逻辑不知道自己正在被 CLI、Telegram 还是 API 调用。

2. **Memory 是一等公民**
   「越来越懂我」是核心竞争力，Memory 必须从第一天就认真设计。

3. **先跑通链路，再系统化升级**
   先实现可运行的最小闭环，再逐步替换 Mock、加强能力。

4. **取长补短，而不是复制**
   - 从 **Pi** 学习极简核心与 Lazy Skills
   - 从 **Claude Code 分析** 学习分层 Memory 与权限
   - 从 **Grok Build** 学习模块划分与 Headless 思想

5. **本地优先，可扩展到多端**

## 整体架构

```text
┌─────────────────────────────────────────────────┐
│                  Channels 层                     │
│  CLI · Desktop · REST API · Telegram · WeChat   │
└──────────────────────┬──────────────────────────┘
                       │ 统一消息协议
                       ▼
┌─────────────────────────────────────────────────┐
│                   Core Harness                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │  Agent   │  │  Memory  │  │    Skills    │  │
│  │  Loop    │  │ 分层记忆  │  │  Lazy Skills │  │
│  └────┬─────┘  └────┬─────┘  └──────┬───────┘  │
│       │             │               │          │
│       └─────────────┼───────────────┘          │
│                     ▼                          │
│            ┌─────────────────┐                 │
│            │  Tools + 权限    │                 │
│            └─────────────────┘                 │
└─────────────────────────────────────────────────┘
```

**核心模块职责**：

- **Agent Loop**：思考与工具调用的循环
- **Memory**：分层记忆（Working / Session / Long-term / Archival）
- **Skills**：按需加载的能力扩展
- **Tools**：具体执行能力 + 权限控制
- **Channels**：各种接入方式，全部适配到统一协议

## 技术选型

- **语言**：TypeScript
- **运行时**：Node.js ≥ 20
- **包管理**：pnpm（Monorepo 友好）
- **模块化**：npm / pnpm workspaces
- **后续考虑**：VitePress（文档）、Electron/Tauri（桌面端）

## 开发节奏

我们采用「阶段式」推进：

1. 最小可用核心（Agent + Tool + MockLLM）
2. 分层 Memory
3. 交互式 CLI
4. 代码清理与重构
5. 真实 LLM 接入
6. Skills 系统
7. 持久化与多渠道
8. 权限与安全

每一阶段结束后，代码都应处于可运行状态。
