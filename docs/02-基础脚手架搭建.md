# 02. 基础脚手架搭建

## 目标

搭建一个清晰、可扩展的 TypeScript Monorepo 结构，为后续所有功能打下基础。

## 最终目录结构

```text
hachimi/
├── packages/
│   ├── core/                 # 核心 Harness（与渠道无关）
│   ├── shared/               # 公共工具
│   ├── channels/             # 各种接入渠道
│   ├── tools-builtin/        # 内置工具
│   └── storage/              # 存储实现
├── apps/
│   ├── cli/
│   ├── desktop/
│   └── server/
├── scripts/                  # 临时脚本与测试
├── config/
├── docs/
├── package.json
└── tsconfig.json
```

## 初始化项目

```bash
mkdir hachimi
cd hachimi
pnpm init
```

配置 `package.json` 支持 workspaces：

```json
{
  "name": "hachimi",
  "private": true,
  "type": "module",
  "workspaces": [
    "packages/*",
    "packages/channels/*",
    "packages/tools-builtin/*",
    "packages/storage/*",
    "apps/*"
  ],
  "scripts": {
    "typecheck": "tsc --noEmit"
  },
  "devDependencies": {
    "typescript": "^5.8.2",
    "@types/node": "^22.13.10"
  }
}
```

## 创建核心包

### packages/core

这是整个项目最重要的包，所有 Agent 逻辑都在这里。

```bash
mkdir -p packages/core/src/{agent,memory,tools,skills,session,permissions,hooks,types}
```

`packages/core/package.json` 示例：

```json
{
  "name": "@hachimi/core",
  "version": "0.1.0",
  "private": true,
  "type": "module",
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "exports": {
    ".": "./src/index.ts"
  },
  "dependencies": {
    "@hachimi/shared": "workspace:*"
  }
}
```

### packages/shared

用于放置 `generateId` 等公共工具函数。

## TypeScript 配置

根目录 `tsconfig.json`：

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "declaration": true,
    "outDir": "dist",
    "baseUrl": ".",
    "paths": {
      "@hachimi/core": ["packages/core/src"],
      "@hachimi/shared": ["packages/shared/src"]
    }
  },
  "include": ["packages/**/*", "apps/**/*", "scripts/**/*"]
}
```

## 为什么采用这种结构？

- **core 完全渠道无关**：这是最重要的设计决策之一
- **packages 按职责拆分**：方便后期独立演进和测试
- **apps 只做组装**：真正的业务逻辑尽量下沉到 packages
- **便于教程分阶段讲解**：每一阶段只动少数目录
