# 06. 代码重构实践

## 为什么需要重构？

在快速实现最小闭环的过程中，我们不可避免地产生了一些「临时代码」：

- `generateId` 在多个文件中重复定义
- 类型定义出现重复（`MemoryEntry`）
- 导入路径混乱
- Monorepo 的 workspace 配置不正确
- CLI 中遗漏了正常对话处理逻辑

这些在早期能跑通没问题，但如果不及时清理，会随着功能增加迅速恶化。

本阶段我们专门停下来，对已有代码做了一次系统性整理。

## 重构原则

我们遵循几个简单原则：

1. **只重构已经跑通的部分**，不借机大改架构
2. **优先解决重复和混乱**，而不是追求完美抽象
3. **保持功能行为不变**（重构前后测试用例都要能通过）
4. **为后续扩展做准备**，而不是过度设计

## 主要重构内容

### 1. 统一 generateId

**问题**：`generateId` 在 `agent.ts`、`llm.ts`、`memory/manager.ts` 中都有一份实现。

**解决**：

- 把 `generateId` 提取到 `@hachimi/shared`
- 所有地方统一从 shared 导入

```ts
// packages/shared/src/index.ts
export function generateId(prefix = ""): string {
  return `${prefix}${crypto.randomUUID()}`;
}
```

### 2. 类型定义去重

**问题**：`MemoryEntry` 同时存在于 `src/types/index.ts` 和 `src/memory/types.ts`。

**解决**：

- 把 `MemoryEntry` 和 `MemoryLayer` 作为核心公共类型，放在 `src/types/index.ts`
- `MemorySearchOptions` 作为 Memory 模块内部类型，保留在 `memory/types.ts`
- `MemoryManager` 从公共类型导入

这符合「被多个模块使用的类型放公共位置，模块私有类型放自己目录」的原则。

### 3. 修正 Monorepo 配置

**问题**：pnpm 无法正确识别本地包，出现 `ERR_PNPM_FETCH_404`。

**关键修复**：

- 新增 `pnpm-workspace.yaml`：

```yaml
packages:
  - 'packages/*'
  - 'packages/channels/*'
  - 'packages/tools-builtin/*'
  - 'packages/storage/*'
  - 'apps/*'
```

- 本地依赖统一使用 `workspace:*` 协议：

```json
"dependencies": {
  "@hachimi/core": "workspace:*",
  "@hachimi/shared": "workspace:*"
}
```

- 删除 `package.json` 中无效的 `"workspaces"` 字段（pnpm 不使用它）

### 4. 修复 CLI 正常对话逻辑缺失

在完善命令处理时，不小心把调用 `agent.run()` 的代码删掉了，导致所有正常输入都无法进入 Agent。
修复后，CLI 的处理顺序变为：

1. 特殊命令（`/memories`、`/remember`、`/exit` 等）优先拦截
2. 其余输入全部交给 `agent.run()`

这是一个典型的「重构时引入的回归问题」，也提醒我们重构后必须做回归测试。

### 5. 其他清理

- 统一导入路径风格
- 删除不再使用的临时注释和调试代码
- 明确 `data/` 目录加入 `.gitignore`

## 重构后的收益

- 代码重复明显减少
- 类型来源单一，维护成本降低
- Monorepo 依赖关系清晰，安装不再报错
- CLI 行为符合预期
- 为后续接入真实 LLM 和 Skills 打下更干净的基础

## 经验总结

- **最小闭环优先，但要及时还债**
  先跑通再重构是对的，但重构不能无限延期。
- **重构也要有测试**
  每次改完都重新跑一遍核心流程（对话、记住、查看记忆、持久化）。
- **工具链问题要尽早解决**
  workspace 配置这类问题越晚处理，影响面越大。
- **保持接口稳定**
  本次重构几乎没有改变对外接口，主要是内部整理，所以风险可控。

## 本章总结

通过这一轮重构，hachimi 的代码从「能跑」变得「更干净、更可维护」。
接下来我们可以更放心地进入下一阶段：接入真实 LLM，让回答质量得到实质性提升。
